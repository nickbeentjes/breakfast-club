**src/tools/identity-query.ts** — `identity_query` tool (MCP-03):

The function `registerIdentityQueryTool(server: McpServer)` registers the tool. The tool:

1. Accepts required `question` parameter (string, description: "Natural-language question about the owner's identity") via Zod schema
2. Accepts optional `limit` parameter (number, default 5, min 1, max 10)
3. Embeds the question using `embedText(question)` from `../embed.js`
4. Runs a `$vectorSearch` aggregation pipeline on the `identity` collection:
   ```typescript
   const results = await db.collection("identity").aggregate([
     {
       $vectorSearch: {
         index: "identity_vector_index",
         path: "embedding",
         queryVector: queryEmbedding,
         numCandidates: 50,
         limit: limit,
         filter: { doc_type: "identity" },
       },
     },
     {
       $project: {
         _id: 0,
         section: 1,
         content: 1,
         sensitivity: 1,
         score: { $meta: "vectorSearchScore" },
       },
     },
   ]).toArray();
   ```
5. Formats results as a readable text response:
   ```
   ## Identity Search Results for: "{question}"

   ### {section} (relevance: {score.toFixed(3)}, sensitivity: {sensitivity})
   {JSON.stringify(content, null, 2) — or a formatted summary}

   ---
   ```
6. Returns `{ content: [{ type: "text", text: formattedResults }] }`
7. If no results found, returns a message: "No identity data found matching your question."

Key points:
- Pre-filter `{ doc_type: "identity" }` ensures future memory chunks (Phase 4) are excluded
- The `numCandidates: 50` is the ANN search breadth; `limit` controls returned results
- Only `console.error()` for logging
- The vector index name `identity_vector_index` must match what was created in Atlas UI (Plan 01-02)

**src/tools/verify-integrity.ts** — `verify_integrity` tool (MCP-05):

The function `registerVerifyIntegrityTool(server: McpServer)` registers the tool. The tool:

1. Accepts optional `repo` parameter (string, default: "breakfast-club-identity", description: "GitHub repository name")
2. Gets the current git tree SHA using one of:
   a. **Preferred:** Execute `git rev-parse HEAD` via `child_process.execSync` (works when MCP server runs from within the repo)
   b. **Fallback:** If git command fails, return an error message explaining the repo is not accessible
3. Returns a formatted response:
   ```
   ## Integrity Check

   **Git Tree SHA:** {sha}
   **Repository:** {repo}
   **Timestamp:** {ISO 8601 now}
   **Attestation Chain:** Pending — attestation log verification will be available in Phase 3

   Status: SHA retrieved successfully. Cross-verification with public attestation chain is not yet implemented.
   ```
4. Does NOT error when attestation chain is unavailable — returns a human-readable note per RESEARCH.md recommendation
5. Returns `{ content: [{ type: "text", text: formattedResult }] }`

Use `import { execSync } from "node:child_process";` for the git command. Wrap in try/catch — if git fails, report the error cleanly.
