**src/index.ts** — MCP server entry point following RESEARCH.md Pattern 1:

```typescript
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import { registerIdentityContextTool } from "./tools/identity-context.js";
import { registerProjectsListTool } from "./tools/projects-list.js";

const server = new McpServer({
  name: "breakfast-club-identity",
  version: "1.0.0",
});

// Register tools
registerIdentityContextTool(server);
registerProjectsListTool(server);

async function main() {
  const transport = new StdioServerTransport();
  await server.connect(transport);
  console.error("Breakfast Club MCP server running on stdio");
}

main().catch((error) => {
  console.error("Fatal error:", error);
  process.exit(1);
});

// Graceful shutdown
process.on("SIGINT", () => {
  console.error("Shutting down...");
  process.exit(0);
});
```

Key points:
- All imports use `.js` extension (required for Node16 module resolution with ESM)
- Only `console.error()` — stdout is reserved for JSON-RPC (MCP-07)
- Tool registration is delegated to per-tool modules that accept the McpServer instance
- SIGINT handler for graceful shutdown
- Will add identity_query and verify_integrity tool registrations in Plan 01-04

**src/tools/identity-context.ts** — `identity_context` tool (MCP-02):

The function `registerIdentityContextTool(server: McpServer)` registers the tool. The tool:

1. Accepts optional `max_tokens` parameter (number, default 1200) via Zod schema
2. Calls `getDb()` to get the MongoDB connection
3. Queries the `identity` collection for all documents with `doc_type: "identity"`
4. Assembles a synthesized context string from persona, skills, and projects sections. Format:

```
## Identity Context

### Persona
Name: {name}
Style: {communication_style.primary}
Approach: {working_style.approach}
Tools: {working_style.tools}
Key strengths: {working_style.strengths joined}

### Skills
Primary stack: {languages, databases, infrastructure joined}
Domain expertise: {domain names with depth levels}

### Active Projects
- {name} ({status}): {description}
- ...
```

5. If assembled text exceeds a rough token estimate (`Math.ceil(text.split(/\s+/).length * 1.3)`), truncate projects list (keep first 3) and trim domain expertise details
6. Returns `{ content: [{ type: "text", text: contextString }] }`

**src/tools/projects-list.ts** — `projects_list` tool (MCP-04):

The function `registerProjectsListTool(server: McpServer)` registers the tool. The tool:

1. Accepts optional `include_completed` parameter (boolean, default false) via Zod schema
2. Calls `getDb()` to get the MongoDB connection
3. Queries the `identity` collection for the document with `{ doc_type: "identity", section: "projects" }`
4. Extracts `content.active` array (and optionally `content.completed`)
5. Formats each project as:
```
- **{name}** [{status}] — {description}
  Stack: {stack joined, if present}
```
6. Returns `{ content: [{ type: "text", text: projectsList }] }`

Both tool handlers:
- Use only `console.error()` for logging
- Import `z` from `zod` for input schema validation
- Import `getDb` from `../db.js`
- Wrap the handler body in try/catch, rethrowing errors (fail loud per CLAUDE.md)
- Accept `McpServer` as parameter (not import it) to keep tool modules testable
