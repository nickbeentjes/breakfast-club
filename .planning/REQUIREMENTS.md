# Requirements: Breakfast Club

**Defined:** 2026-03-26
**Core Value:** Any AI, any provider, reads the same you — without repeating yourself, and without any company owning the relationship you've built.

## v1 Requirements

### Identity Schema

- [x] **IDNT-01**: Private repo template has a defined JSON schema for persona (name, communication style, working style, preferences, dislikes)
- [x] **IDNT-02**: Private repo template has a defined JSON schema for skills (primary stack, domain expertise, trades knowledge)
- [x] **IDNT-03**: Private repo template has a defined JSON schema for active projects (name, status, description, stack)
- [x] **IDNT-04**: Private repo template has a defined JSON schema for values and beliefs (professionally relevant fields only)
- [x] **IDNT-05**: All schema fields have a sensitivity label (personal/professional/public) for projection enforcement
- [x] **IDNT-06**: Founder's identity (nick-identity.json) is loaded as seed data and validates against the schema

### MCP Server — Owner Tools

- [x] **MCP-01**: MCP server runs as stdio process compatible with Claude Desktop and Claude Code
- [x] **MCP-02**: `identity_context` tool returns synthesized persona/skills/projects context for system prompt injection (600-1200 token budget)
- [x] **MCP-03**: `identity_query` tool does semantic search across identity data using MongoDB Atlas vector search
- [x] **MCP-04**: `projects_list` tool returns current active projects with status
- [x] **MCP-05**: `verify_integrity` tool returns current git tree SHA and checks against public attestation chain
- [x] **MCP-06**: MCP server connects to MongoDB lazily on first tool call and caches the connection for the session lifetime
- [x] **MCP-07**: All MCP server logging goes to stderr (stdout reserved for JSON-RPC message stream)

### Projection Engine

- [ ] **PROJ-01**: Projection system uses whitelist model — only explicitly listed field categories are included in a projection
- [ ] **PROJ-02**: At least three built-in projections: personal (full), professional (skills/projects/work-style), public (summary only)
- [ ] **PROJ-03**: Projection enforcement happens server-side in the MCP tool handler, never client-side
- [ ] **PROJ-04**: Owner can create custom projections as JSON files in the private repo's `projections/` directory
- [ ] **PROJ-05**: Projection tokens are scoped API keys — a token tied to a projection can only access that projection's data

### Recruiter Chatbot

- [ ] **RCTR-01**: A chatbot endpoint exists that accepts a scoped token and serves only the professional projection
- [ ] **RCTR-02**: Chatbot system prompt constrains it to role-fit, experience, and logistics questions (salary and personal data explicitly out of scope)
- [ ] **RCTR-03**: Recruiter can ask natural language questions and receive contextual answers grounded in real identity data
- [ ] **RCTR-04**: Chatbot is queryable via a simple web UI (shareable URL with token) requiring no technical setup by the recruiter
- [ ] **RCTR-05**: Every recruiter query is logged in an append-only audit trail in the private repo

### Sync Pipeline

- [ ] **SYNC-01**: GitHub Action triggers on push to the private repo
- [ ] **SYNC-02**: Action reads identity JSON files using the Git Trees API (single call, not per-file fetches)
- [ ] **SYNC-03**: Action computes SHA-256 of canonical JSON (RFC 8785 JCS) as `source_hash` and stores it in MongoDB with each document
- [ ] **SYNC-04**: Action stores the git tree SHA alongside the canonical hash for the verification triangle
- [ ] **SYNC-05**: Cross-repo auth uses a GitHub App (not PAT) with Contents:Read on private repo and Contents:Read+Write on public repo
- [ ] **SYNC-06**: Third-party Actions in the sync workflow are pinned to commit SHAs, not version tags
- [ ] **SYNC-07**: Bot commits to the public repo include `[skip ci]` to prevent workflow loops

### Transparency and Verification

- [ ] **VERF-01**: Public GitHub repo contains an append-only hash attestation log (each entry records `source_hash`, `tree_sha`, `timestamp`, `prev_attestation_hash`, `self_hash`)
- [ ] **VERF-02**: GitHub Pages dashboard loads `data/state.json` and `data/chain/{id}.json` (pre-generated by sync Action) and verifies the chain client-side using WebCrypto SubtleCrypto
- [ ] **VERF-03**: Dashboard shows current verification state (verified/tampered/unknown), timeline of updates, and hash consistency
- [ ] **VERF-04**: Dashboard is vanilla HTML/JS with no build step and no server trust required — all verification runs in-browser

### SKILL.md

- [ ] **SKIL-01**: SKILL.md exists in the private repo root with instructions for any AI to use the identity store
- [ ] **SKIL-02**: SKILL.md uses Markdown with `<identity>` XML wrapper format for cross-model compatibility
- [ ] **SKIL-03**: SKILL.md instructs the AI on persona loading, project context, memory search, and projection-scoped behavior
- [ ] **SKIL-04**: SKILL.md stays within 600-token budget when injected as a system prompt prefix

### Conversation Memory

- [ ] **MEM-01**: Memory is stored as chunks in MongoDB with chunk types: turn, summary, semantic, entity
- [ ] **MEM-02**: Chunks use 512-token size with 10-15% overlap, using semantic/recursive chunking (not fixed-size)
- [ ] **MEM-03**: Single MongoDB Atlas vector index covers all chunk types with `chunk_type` as a pre-filter (respects M0 one-index limit)
- [ ] **MEM-04**: Retrieval applies recency weighting (30% weight) to prevent stale memories dominating results
- [ ] **MEM-05**: Memory ingestion supports at minimum Claude-exported conversation logs (Markdown format)
- [ ] **MEM-06**: Memory chunks store source AI (claude/chatgpt/gemini), date, and conversation ID for provenance

## v2 Requirements

### Blockchain Anchoring

- **BLKC-01**: Periodic anchor of latest public repo hash to Polygon PoS (~$0.002/tx)
- **BLKC-02**: did:pkh (wallet address = DID) for identity ownership on-chain
- **BLKC-03**: Verifiable Credentials via Veramo for projection access tokens
- **BLKC-04**: ERC-5192 soulbound tokens for identity ownership
- **BLKC-05**: ERC-4907 rental tokens for time-limited recruiter access (auto-expires)

### Advanced Privacy

- **PRIV-01**: ZK proofs (Circom/snarkjs) for credential assertions without revealing data
- **PRIV-02**: SD-JWT (RFC 9901) selective disclosure for recruiter tokens
- **PRIV-03**: BBS+ signatures for unlinkable presentations across employers
- **PRIV-04**: Values alignment computation — internal compatibility assessment against full private identity, surfaces only conclusion (flag) to recruiter
- **PRIV-05**: Per-requester watermarking (invisible Unicode + canary data) for leak detection

### Ecosystem Integrations

- **INTG-01**: ChatGPT custom GPT integration via REST API
- **INTG-02**: Gemini integration
- **INTG-03**: Conversation export automation (API-accessible history where providers allow)
- **INTG-04**: Schema migration tooling for identity version upgrades

### Bittensor Subnet

- **BITT-01**: Subnet design for decentralized memory categorization and integrity verification
- **BITT-02**: Miner implementation for receiving and categorizing AI-generated memories
- **BITT-03**: Validator implementation for hash integrity challenges

## Out of Scope

| Feature | Reason |
|---------|--------|
| Any paid infrastructure in Milestone 1 | Must run at $0/month — free tiers only |
| UI for editing identity data | Git + JSON IS the editing interface; a UI adds complexity without value for v1 |
| Custom authentication system | GitHub access controls are the auth for v1; no need to reinvent |
| ChatGPT custom GPT (v1) | Claude-first for demo; REST API hook is sufficient to enable it later |
| Real-time sync (webhooks to MCP clients) | Pull-on-demand is sufficient for v1; push complexity not justified |
| Multi-user identity sharing | Breakfast Club is a personal identity system, not a team one |
